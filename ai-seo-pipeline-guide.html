Зачем это

Контент — это конвейер, а не один большой текст

Раньше одна статья = один человек делает всё подряд: ищет, пишет, проверяет, оформляет, публикует. Медленно и нестабильно по качеству.

Мы разбили процесс на этапы, и за каждый отвечает свой агент с чёткой инструкцией. Ты как редактор-постановщик: запускаешь нужный шаг, проверяешь результат, идёшь дальше.

Главная идея: каждый шаг оставляет артефакт (research-файл, черновик, Google Doc, WP-черновик). Ничего не теряется, любой этап можно перезапустить отдельно.
Основа

Скилл = агент внутри Claude

Claude Code — это Claude, запущенный в папке проекта (в терминале или IDE). Он видит файлы и умеет выполнять команды.

Обычный чат просто отвечает на вопрос. А скилл — это заранее обученный агент под конкретную задачу: у него прописана инструкция, наш бриф, источники, формат и правила. Вызываешь /researcher — и Claude переключается в роль ресёрчера, а не «универсального помощника».

Обычный чат

  • Отвечает «вообще», без нашего контекста
  • Не знает голос бренда и правила
  • Каждый раз объясняешь заново
  • Результат непредсказуем

Скилл-агент /имя

  • Узкая роль с должностной инструкцией
  • Знает бриф, источники, формат, тон
  • Делает шаг по нашему стандарту
  • Оставляет файл-артефакт для следующего

Проще говоря: скилл — это сотрудник с должностной инструкцией. Конвейер — это отдел из таких сотрудников, которые передают работу по цепочке.

С чего начать

Как запускать через Claude

1. Открыть
Claude Code в папке проекта (напр. ai-seo). Он сразу видит брифы, research, данные клиента.
2. Вызвать
Пишешь команду со слешем: /researcher, /writer и т.д. Можно с уточнением — «/writer тема X, тон строгий».
3. Проверить
Агент отрабатывает и оставляет файл/черновик. Смотришь результат — правишь или идёшь на следующий шаг.
4. Передать
Следующий скилл берёт артефакт предыдущего. Порядок — ниже.
/researcher→/writer→/editor→/fact-checker→/quality-gate→/seo-optimizer→[/designer]→/publisher→/tracker→/page-optimizer
Пайплайн

10 шагов: что делает каждый скилл

Каждая карточка — один агент: что подаёшь на вход, что он делает, что отдаёт.

01
/researcher  Глубокий ресёрч темы
Вход
Бриф / тема
Делает
4 субагента-разведчика (аналитик, контрарианец, практик, живая боль) собирают 30+ источников, верифицируют цифры, проверяют каннибализацию.
Выход
Research Report + зафиксированный план H2
02
/writer  Сильный копирайтер
Вход
Research-файл + бриф
Делает
Пишет живой черновик: голос компании, ритм русского текста, семантика. Не ИИ-«вода».
Выход
Черновик статьи + черновой SEO-блок
03
/editor  Редактура
Вход
Черновик (или Google Doc ID)
Делает
Убирает ИИ-шность, добавляет голос бренда и конкретику вместо расплывчатости. По Google Doc — правит и оставляет комментарии.
Выход
Отредактированный текст
04
/fact-checker  Проверка фактов
Вход
Текст / Google Doc
Делает
Верифицирует факты и цифры по уровням T1/T2/T3. Сомнительное помечает, источники — только реальные.
Выход
Текст без выдуманных данных
05
/quality-gate  Авто-чеклист качества
Вход
Готовый текст
Делает
19 метрик PASS/FAIL: язык, структура, E-E-A-T, наличие кейса и позиции бренда. Гейт перед публикацией.
Выход
Вердикт PASS / список правок
06
/seo-optimizer  SEO-упаковка
Вход
Финальный текст
Делает
H1 в 3 вариантах, SEO title, description, обложка, проверка H2 как запросов, FAQ из Wordstat, внутренние ссылки.
Выход
SEO-пакет статьи
07
[/designer]  Инфографика (опц.)
Вход
Статья
Делает
Находит 2–3 места под схемы, генерит в стиле клиента (Pencil или HTML→Playwright), грузит PNG в WP и вставляет.
Выход
Статья с инфографикой
08
/publisher  Публикация
Вход
Готовая статья
Делает
Создаёт Google Doc на ревью + черновик в WP одновременно. Ждёт «публикуем» → ACF, обложка, типографика, publish.
Выход
Опубликованная статья
09
/tracker  Мониторинг
Вход
Опубликованный URL
Делает
Снимает позиции (Вебмастер) и AI-упоминания (keys.so, Алиса/GPT). Даёт action items, обновляет llms.txt.
Выход
Отчёт + задачи на доработку
10
/page-optimizer  Дожим страниц
Вход
URL + action items
Делает
Точечно усиливает живую страницу: новый H2, FAQ+Schema, внутренние ссылки, переобход. Без переписывания.
Выход
Улучшенная страница + переобход
Вне основного конвейера

Оркестраторы и вспомогательные

Нужны не для каждой статьи, а на старте проекта или для аудитов.

/marketing-audit
Оркестратор аудита сайта: запускает по очереди тех-SEO → контент/семантику → конкурентов и gap. Для нового клиента или диагностики.
/knowledge-session
Структурированное интервью с клиентом (~60 мин, ~20 вопросов). Выдаёт company-context.md и voice.md — на них работают writer/editor.
/seo-tech-audit
Тех. аудит: robots, sitemap, индексация, мета, canonical, скорость, структура URL.
/seo-content-audit
Семантическое ядро, кластеризация по интентам, покрытие страницами, дыры в контент-плане.
/seo-competitors-gap
Находит органических конкурентов через keys.so и gap-анализ — какие ключи есть у них и нет у нас.
/audit-report-builder
Собирает из research-файлов единый HTML-отчёт для клиента — с навигацией и summary.
Доступы

Подключение: Google Docs + WordPress

Чтобы /publisher создавал Google Doc на ревью и черновик в WP, нужны два подключения.

📝 WordPress

  1. URL сайта — адрес WordPress
  2. Логин + Application Password — спец-пароль для REST API (Пользователи → профиль → Application Passwords)
  3. Тип записи — куда публикуем (напр. blog)
  4. ACF author ID — id автора для поля статьи
Даёт права публиковать через /wp/v2/. Application Password, не основной пароль.

📄 Google Docs

  1. Google Cloud проект с включёнными Google Docs + Drive API
  2. OAuth-клиент со scopes documents + drive
  3. token.json — авторизация для MCP-сервера google-docs
⚠️ Перевести OAuth consent screen в Production, иначе refresh-токен умирает раз в 7 дней (постоянный invalid_grant).
Безопасность: сами пароли и токены — в приватной заметке, не в этой презентации. Здесь только перечень, какие данные нужны и где их взять.
